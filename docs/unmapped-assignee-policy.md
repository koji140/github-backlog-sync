# assigneeMap 未定義ユーザーの運用ポリシー

## 1. 背景

GitHub Projects の assignee（担当者）を Backlog の担当者へ同期するには、
`config/mapping.json` の `assigneeMap` に GitHub ユーザー名 → Backlog ユーザー ID の対応を手動で定義する必要がある。

しかし、すべての GitHub ユーザーが `assigneeMap` に登録されているとは限らない。
新メンバー追加時・外部コントリビューター・ゲストアカウントなど、未定義ユーザーが発生しうる。

大野さん・潮田さんのような既知メンバーを `assigneeMap` に追加する作業を行う前に、
**未定義ユーザー全般に適用できる運用ルール** を先に決める必要がある。

---

## 2. このドキュメントの位置づけ

- 個別メンバーを `assigneeMap` に追加するための **前提となる運用ルール** を定義する
- 「先にルールを決め、その後で個別定義を追加する」という順序を守るための設計記録
- コード実装の前に方針を合意しておくことで、後から仕様変更が発生しないようにする

---

## 3. 現状コードの挙動（Phase 1 時点）

`mapGitHubItemToBacklogIssue()` 内の処理：

```
GitHub assignee ログインが assigneeMap に存在しない
  → assigneeId = null
  → [Map] 担当者マッピングなし: GitHub ユーザー "xxx" は assigneeMap に存在しません。 （警告ログ）
```

`buildBacklogUpdateParams()` 内の処理：

```
currentAssigneeId（Backlog 既存）≠ payload.assigneeId（null）
  → 差分あり と判定
  → params.assigneeId = null を diff に含める
  → updateBacklogIssue() が assigneeId='' を送信
  → Backlog の既存担当者がクリアされる ← 問題
```

**つまり現状は「警告ログ + Backlog 既存担当者のクリア」という挙動であり、
意図した warn-only（警告のみ・既存担当者を維持）にはなっていない。**

また、現コードは以下の 2 ケースを区別できない：

| ケース | GitHub 状態 | assigneeId |
|--------|------------|-----------|
| A | GitHub に担当者なし（意図的な未設定） | null |
| B | GitHub に担当者あり・但し未マッピング | null |

ケース A では Backlog をクリアしてよいが、ケース B ではクリアすべきでない。
この区別を実装するにはコード変更が必要（後述）。

---

## 4. 選択肢

### オプション 1: warn only（警告のみ）

未定義ユーザーがいても同期を継続する。
`assigneeId` を送らず、Backlog 側の既存担当者を変えない。
警告ログで検知する。

### オプション 2: default assignee（デフォルト担当者）

未定義ユーザーの場合、`mapping.json` で指定したデフォルト担当者を使う。

### オプション 3: fail-fast（エラー停止）

未定義ユーザーが 1 件でもいたら同期を中断してエラーを返す。

---

## 5. 選択肢の比較

| 観点 | warn only | default assignee | fail-fast |
|------|-----------|-----------------|-----------|
| 安全性 | 高（既存を変えない） | 中（誤担当者が入るリスク） | 高（確実に検知） |
| 誤更新リスク | 低 | 中〜高 | なし（止まる） |
| 初期導入しやすさ | 高（即使える） | 中（デフォルト設定が必要） | 低（全員マッピング必須） |
| 運用負荷 | 低（ログ確認のみ） | 低（設定次第） | 高（マッピング漏れで止まる） |
| 担当者同期の精度 | 低（未定義は無視） | 中（近似値） | 高（完全一致が保証される） |

---

## 6. 暫定方針（Phase 1）

**warn only を採用する。**

理由：

- 初期導入フェーズでは全ユーザーのマッピングが揃っていないことが多い
- fail-fast は運用開始を阻害する
- default assignee は誤った担当者が大量作成されるリスクがある
- Backlog 既存担当者を勝手に変えない方が安全

### warn only の定義

- `assigneeMap` に存在しない GitHub ユーザーがいても同期は継続する
- `assigneeId` は送らない（Backlog 側の担当者を変更しない）
- Backlog 側に既存の担当者がいる場合は、そのまま維持する
- 警告ログ `[Map] 担当者マッピングなし` で検知できるようにする

### 現状コードとの差分（要実装）

現状コードは「未マッピング = assigneeId: null = 差分あり → クリア」となっており、
warn only の定義を満たしていない。

完全な warn only を実現するには、以下の変更が必要：

1. `mapGitHubItemToBacklogIssue()` で、未マッピング時に `assigneeId: null` ではなく
   「未マッピングであることを示す専用値（例: `undefined` または `'UNMAPPED'`）」を返す
2. `buildBacklogUpdateParams()` で、`payload.assigneeId` が未マッピング値の場合は
   assigneeId を diff 比較の対象から除外する

この実装は「今後の作業」として別途対応する（本ドキュメントのスコープ外）。

---

## 7. この方針の意味

- **先にルールを決め、その後で個別ユーザーを定義する**
- 大野さん・潮田さんなど既知メンバーの追加は、このルールが合意された後に行う
- `assigneeMap` に追加された既知ユーザーは正確に同期される
- 未定義のユーザーは warn only で安全に無視される
