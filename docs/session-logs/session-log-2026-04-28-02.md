## 今日やったこと（詳細）

**テーマ**: `sync-backlog-to-github.js` に担当者（assignee）同期を追加する

### 作業内容

1. **設計確認**
   - 現状: Backlog→GitHub 同期はステータスのみ
   - 追加要件: Backlog で担当者を変更したら GitHub にも反映したい
   - 設計方針を決定:
     - `assigneeMap`（GitHub login → Backlog userId）を自動逆引きして `reverseAssigneeMap` を生成
     - Backlog 担当者が null（未設定）の場合は GitHub をクリアしない（スキップ）
     - マッピングにない Backlog userId は警告ログのみ出力してスキップ
     - 担当者更新は GitHub Issues REST API（`PATCH /repos/{owner}/{repo}/issues/{n}`）を使用
     - ステータスと同じ 5分閾値で動作（last-modified wins）

2. **実装（`scripts/sync-backlog-to-github.js`）**
   - `buildReverseAssigneeMap()`: assigneeMap から逆引きマップを生成
   - `parseGitHubIssueUrl()`: sync metadata の GitHub Issue URL から owner/repo/number を解析
   - `updateGitHubIssueAssignees()`: GitHub Issues REST API で assignees を更新
   - main() の分類ループに担当者マッピングを追加（`targetGitHubLogin` / `assigneeUnmapped`）
   - 実行ループで担当者更新を追加（ステータス更新の直後）
   - プランログに担当者情報を追加

3. **設定ファイル更新**
   - `config/mapping.example.json`: `reverseAssigneeMap` セクションを追加（通常は空でOK）

4. **ドキュメント更新**
   - `CLAUDE.md`: `reverseAssigneeMap` の生成ルールと担当者更新の仕組みを追記
   - `README.md`: 同期方向の説明を「ステータス・担当者」に更新

5. **コミット & プッシュ**

## 石丸の思考プロセス

- 「担当者もBacklog→GitHubで連携したい」という要件から、既存の reverseStatusMap の設計を参考にそのまま reverseAssigneeMap を設計した
- ステータスと担当者で異なるAPIを使う（GraphQL mutation vs REST PATCH）ことに注意
- Backlog null → GitHub クリアは危険（意図せず全担当者が消える可能性）なのでスキップにした
- mapping.json の `assigneeMap` が既に正しく設定されていれば `reverseAssigneeMap` の追加設定は不要（自動生成）

## うまくいったこと

- sync metadata に GitHub Issue URL が既に入っていたので、URL解析だけで REST API に必要な情報が揃った
- DRY_RUN のプランログにも担当者情報が出るようにしたので、実行前の確認がしやすい
