# 概要・背景

## 背景

プロジェクト管理ツールとして GitHub Projects と Backlog を並行して使用しているチームでは、
どちらかを正本とした同期の仕組みがないと、以下の問題が生じます。

- 同じ情報を 2 箇所に手動入力する二重管理が発生する
- GitHub を使わないメンバーが最新状況を把握できない
- ステータス更新の反映漏れ・遅延が起きやすい

## なぜこの連携を作るのか

- エンジニアは GitHub Projects を主軸に作業管理を行っている
- 非エンジニアメンバーやステークホルダーは Backlog での進捗確認を好む
- 手動での転記作業をゼロにし、情報の信頼性を高めたい

**結論**: GitHub Projects を唯一の正本として、Backlog を読み取り専用の反映先として扱う。

## 対象スコープ（初期フェーズ）

| 対象 | 内容 |
|------|------|
| 同期元 | GitHub Projects（GraphQL API v2 経由） |
| 同期先 | Backlog プロジェクト（REST API 経由） |
| 同期項目 | タイトル、説明、ステータス、担当者 |
| 実行方法 | 手動トリガー（workflow_dispatch）+ 定期実行（schedule） |
| 言語 | Node.js（ESM） |

## 非対象スコープ

以下は初期フェーズでは対象外とし、必要に応じて後続フェーズで対応します。

- Backlog → GitHub の逆方向同期
- コメントの同期
- 添付ファイルの同期
- リアルタイム（webhook）トリガーによる即時反映
- Pull Request の同期
- GitHub Issue に直接紐づかない Projects アイテム（Draft Issue）の同期

## 今後の拡張方針

1. **フェーズ 2**: 実装完成 + GitHub Secrets による本番運用開始
2. **フェーズ 3**: 同期済みアイテムの追跡（重複作成防止、差分更新）
3. **フェーズ 4**: Webhook によるリアルタイム同期
4. **フェーズ 5**: 双方向同期の検討（競合解決ルールの設計が必要）
5. **将来**: コメント・添付ファイル同期の検討
